# dev-workflow - 开发工作流程

> **定位：** {{PROJECT_NAME}} 的「批注循环 + 放手执行」开发方法论

---

## 一、核心原则

### 1.1 单向推进

- 严禁缺少 Plan 直接写代码
- 先有想法，别上来就问
- 拿不准时，先问再动

### 1.2 隔离开发

- 模块化逐个击破
- 当前模块未通过测试前，绝不开启下一模块修改

## 三、详细流程说明

### 3.1 Step 1: 调研 Research

- **目标**：深入理解现有代码库
- **产出**：`plans/research/research-{功能名/v{版本号}}.md`
- **命名**：有版本号用 `research-v{版本号}.md`，想法/探索用 `research-{功能名}.md`
- **关键**：用户先看完再往下

### 3.2 Step 2: 规划 Planning

- **产出**：`plan-{功能名}-v{版本}.md`，包含：
  - 代码片段 + 文件路径 + 技术权衡
  - **Skill 声明**（必须）：列出实施时需加载的 skill

> Plan 制定规范详见 [planner.md](planner.md)。

### 3.3 Step 3: 批注循环（核心）

直接在 plan.md 里加批注，1-6 轮直到计划完全没问题：

| 批注标记 | 类型 | 说明 |
|---------|------|------|
| 🔴 | 批注/否决 | 纠正错误假设、否决不合理方案 |
| 🟡 | 补充 | 添加技术或业务约束 |
| 🔵 | 流程图 | 画流程图解释逻辑 |
| ⏸️ | don't implement yet | 暂不实施，不影响整体状态推进 |

每条批注必须有决策（✅ 采纳 / ❌ 不同意 / ✏️ 调整 / ⏭️ 跳过）才能推进，决策后立即原地清理。

> 完整批注规则见 [annotater.md](annotater.md)。

### 3.4 Step 4: 放手执行

**前置 1：原地清理 plan.md**
- 采纳内容融入正文，删除批注标记
- 删除 `⏸️ don't implement yet` 内容
- 状态改为 `实施中`

**前置 2：加载 Skill**（强制，不加载 = 盲写必返工）

**执行指令**：

| 指令 | 说明 |
|------|------|
| `implement it all` | 全部实现 |
| `mark completed` | 每完成一项标记 |
| `don't stop` | 一口气做完 |

### 3.5 Step 5: 构建 + 测试

根据项目实际命令执行构建和测试验证，根据结果修复问题。

### 3.6 Step 6: Review 收尾

创建 `review-{功能名}-v{版本}.md`：

| 章节 | 内容 |
|------|------|
| 验收总览 | 功能完成度、构建/测试/运行三板斧 |
| Plan 偏差记录 | 计划与实际的偏差 |
| 遗留问题 | P0/P1/P2 分级管理 |
| 验收检查项 | 对照 plan 检查点逐项验收 |
| 经验总结 | 踩坑 → MEMORY.md |

## 四、Review 问题分级与签收

### 4.1 问题分级

| 级别 | 说明 | 处理方式 |
|------|------|---------|
| **P0** | 阻塞，功能无法使用 | 必须修复才能签收 |
| **P1** | 重要，影响核心功能 | 必须关闭或转入新 plan 才能签收 |
| **P2** | 次要，不影响签收 | 可遗留，但需注明 |

### 4.2 签收条件

- ✅ 所有 P0 已关闭
- ✅ 所有 P1 已关闭或已转入新 plan
- ⚠️ P2 可遗留，但需在 review 中注明

### 4.3 双向同步

review 是问题的 source of truth，plan 只挂引用：

1. review 遗留问题表记录（主记录）
2. 同步回 plan 阶段跟踪表（状态改为 ⚠️ review 打回）
3. 修复后双向更新：review ✅ 已修复、plan ✅ 已完成

## 五、Plans & Reviews 系统

### 5.1 目录结构

```
plans/
├── research/                          # 调研文档
│   ├── research-v{版本号}.md          # 与版本绑定
│   └── research-{功能名}.md           # 想法/探索阶段
├── template/                          # 模板
│   └── plan-template.md
├── plan-{功能名}-v{版本}.md           # 带版本的计划
└── plan-{功能名}.md                   # 独立功能/改造

reviews/
├── template/
│   └── review_template.md
├── review-{功能名}-v{版本}.md         # 功能收尾（必须）
└── code-review-{功能名}-v{版本}.md    # 代码审查（按需）
```

### 5.2 Plan 命名约定

| 格式 | 示例 | 说明 |
|------|------|------|
| `plan-{功能名}-v{版本}.md` | `plan-dashboard-v1.0.md` | 带版本迭代 |
| `plan-{功能名}.md` | `plan-error-handler-improvement.md` | 独立功能 |

### 5.3 Plan 状态生命周期

```
┌──────────┐    批注完成     ┌──────────┐    执行完成    ┌───────────┐
│  规划中   │ ──────────────► │  实施中   │ ────────────► │ review 中  │
└──────────┘                 └──────────┘               └─────┬─────┘
                                                             │
                                               ┌─────────────┴─────────────┐
                                               │                           │
                                      有 P0/P1 问题                P0/P1 已关闭
                                               ▼                           ▼
                                        ┌───────────┐               ┌──────────┐
                                        │ review 打回 │               │  已完成   │
                                        └─────┬─────┘               └──────────┘
                                              │                          ▲
                                     修复完成后重新 review ─────────────────┘
```
